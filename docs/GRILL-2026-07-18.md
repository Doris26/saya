## 总判定: **FAIL**(P0=3 未解决)

理由:PLAN v1.1 的技术选型主线(SwiftPM+CLT、Carbon 热键、剪贴板注入、legacy keychain)经 spike 实测立得住,但存在三个 P0 级结构缺陷:(1) **转写文本不可恢复丢失链**——粘贴成功与否在 macOS 上根本不可探测(CGEventPost 是 fire-and-forget),而 plan 的剪贴板恢复(300ms)+ 转写成功即删音频 + 无历史记录三条规则叠加后,任何一次静默注入失败(重映射 ⌘V / Secure Input / Electron 拦截,§7#1 自己列为"高概率")都把用户口述的全部文字彻底销毁,且 §7#1 承诺的"注入失败自动兜底"因失败不可探测而永远无法触发;(2) **secure-field 拒注与剪贴板兜底的组合泄密**——§2.2 说 secure field 拒绝注入,§7#1 把 secure field 列进"注入失败→自动复制到剪贴板"的类别,自然的错误处理实现(拒注 throw → 通用 catch → 兜底)会把口述的密码广播到 general pasteboard;(3) **测试链整体失效**——本机 CLT 下 XCTest module 不存在(`xcrun --find xctest` 实测失败),§5.1 规划的三个 XCTest 测试文件一个都编译不过,`swift test` 本身救不了它,必须整体改用 swift-testing。三条都是 plan 结构级修订,不是实现细节,故 FAIL;修完这 3 条 + 落实 P1 清单即可转 PASS。

## P0/P1 详细条目(证据 + 修复 + 验收方式)

### #1 [P0] 转写文本不可恢复丢失链(注入静默失败 = 全部口述销毁)
- **证据**:CGEventPost 无返回通道告知目标 App 是否消费了 ⌘V(fire-and-forget);§2.2 L100 "300ms 固定延迟后恢复原剪贴板";§2.3 L110 "转写成功后删除";§0 L24 历史记录=非目标;§7#3 只在**转写**失败时保音频——注入失败时音频已删。§7#1 列的失败场景(Electron 拦截、终端重映射 ⌘V、secure field)自评概率"高"。Wispr Flow/superwhisper/MacWhisper 全部保留 last-transcript 入口,正是因为这个不可探测性。
- **修复**:(a) AppCoordinator 内存保留最近一次 transcript,菜单栏加"复制上次转写"单项(单条不违反 §0 无历史范围,§0 加一句例外注);(b) 音频文件改为**下次录音开始时**才删(始终留一次完整 transcribe+inject 重试机会),隐私由 #26 的启动清扫+0600 兜底;(c) §7#1 重写:粘贴失败**不可探测**,兜底语义='转写始终可从菜单找回',删掉"自动检测失败"措辞。
- **验收**:M3 加一行——在某目标 App 里解绑/拦截 ⌘V,完成一次口述,验证文字仍可从菜单找回(agent 可用两进程 harness 模拟不消费 ⌘V 的接收方)。

### #2 [P0] secure-field 拒注 × 剪贴板兜底 = 密码广播
- **证据**:§2.2 L102 "secure field 时拒绝注入并提示" vs §7#1 L264 失败类枚举含 "secure field"、缓解="注入失败自动兜底'文本已复制到剪贴板'+通知"。自然实现(拒注 throw → coordinator 通用 catch → 兜底)组合出泄漏;general pasteboard 上的密码会被轮询型剪贴板管理器持久化(常见云同步)。
- **修复**:§2.2 与 §7#1 各加一行:secure 上下文的拒注**不属于**注入失败类;该 transcript 丢弃,或仅在 app 自有窗口展示、由用户显式点击复制。
- **验收**:M3 密码框场景断言:拒注后 pasteboard changeCount 不变、剪贴板内容不含 transcript。

### #3 [P0] XCTest 在 CLT 下不存在,测试支柱整体失效
- **证据**:本机复核:`xcrun --find xctest` → `unable to find utility`;scratchpad/swifttest_probe 含 XCTest 版(报 `no such module 'XCTest'`)与 swift-testing 版(`import Testing` + `@Test`,platforms .macOS(.v14) 下 PASS)对照。§5.1 三个 *Tests.swift 均按 XCTest 风格规划(URLProtocol mock 属 Foundation,不受影响)。
- **修复**:(a) 全部测试文件用 swift-testing;(b) test target 继承 `.macOS(.v14)`(默认 10.13 会破 @Test 宏);(c) 拆 `AIVoiceInputCore` library target(Modules/ 全部)+ thin executable,`@testable import` 干净;(d) §5.2-6 改写。
- **验收**:M0 加一行:`swift test` 带一个占位 @Test 跑 green。

### #4 [P1] SecureEventInput 未处理(检测 + 热键 + 注入三面)
- **证据**:Terminal Secure Keyboard Entry / 密码管理器可使系统进入 secure input,且有卡死不释放的档案(keepassxc #11906);Alfred 官方排障明确 secure input 使全局热键失效;Keyboard Maestro/Espanso 各有整页排障。AX 检测对此无感知——Terminal 的焦点元素不是 secure text field,§2.2 自评 AX 在 Electron/Chromium/终端"差"→ 现有守卫在最需要时 fail-open。`IsSecureEventInputEnabled()`(HIToolbox)零权限可查。
- **修复**:注入前无条件查 `IsSecureEventInputEnabled()`;true → 不 post 事件、不恢复剪贴板、通知点名 culprit(kCGSSessionSecureInputPID→app 名);菜单栏加 secure-input active badge;AX secure-subrole 检查降为次级信号;§2.2 显式写 fail-open 策略行:"AX inconclusive AND secure-input off → proceed"。
- **验收**:M3 加验收行:Terminal 开 Secure Keyboard Entry,期望=拒注+transcript 可找回+通知点名 Terminal(agent 可编程开关该状态测)。

### #5 [P1] 静音/误触无 gate,幻觉文本无人值守注入
- **证据**:whisper 家族静音幻觉("感谢观看"类)有大量档案(openai/whisper discussion #1606、AAAS 报道);gpt-4o-transcribe 有改善报告(zenn.dev daishiro)但也有"长静音可触发幻觉"的相反报告——**对本产品未验证**,SPIKE 5 没测静音。§1.1 L60 管线无条件;5min cap 使整链无人值守发生。
- **修复**:三层零成本 gate:<0.7s 丢弃不调 API;averagePower(已有,L68)全程未过噪声底→"未检测到语音"跳过 API(顺带省 L107 担心的误触钱);cap 触发的停止=stop+transcribe+**notify-only 永不自动注入**。
